Question: A data engineer is tuning a Spark SQL join between a very large fact table and a very small dimension table. The current physical plan uses a sort-merge join, causing heavy shuffling. The engineer wants Spark to replicate the small table to executors so the large table does not need to be shuffled for the join.

Options:
  A: Which function should be applied to the small table to force this behavior?
  B: repartition(small_table)
  C: broadcast(small_table)
  D: coalesce(small_table)
  E: cache(small_table)

Correct Answer: 2

Explanation: Overall explanation Broadcast Hash Join (BHJ) is a performance optimization technique in distributed computing. Standard Join (Shuffle): Both Table A and Table B are sorted and exchanged across the network so that matching keys end up on the same node. This is slow for large data. Broadcast Join: Table B (Small) is copied to all nodes. Table A (Large) stays put. The join happens locally. Spark has a configuration spark.sql.autoBroadcastJoinThreshold (default 10MB). If a table is smaller than this, Spark automatically broadcasts it. However, in this scenario, the table might be slightly larger than the threshold, or the optimizer's statistics might be missing. Explicitly using the broadcast() function overrides the estimates and forces the optimization.

Currently, the pipeline is configured in "Continuous" mode. The engineer observes that the cluster remains active and billable 24/7, even though the data processing finishes in just 15 minutes. Options:
  A: The engineer is reviewing the file arrival history:
  B: Which pipeline execution mode should the engineer select to meet the cost-reduction requirement?
  C: Auto Scaling
  D: Continuous
  E: Triggered
  F: Serverless

Correct Answer: 3

Explanation: Overall explanation: In Databricks Lakeflow Spark Declarative Pipelines (and standard Structured Streaming), there are two primary execution modes that dictate how the compute resources interact with the data stream: Continuous Execution: The compute resources stay up 24/7. The system constantly polls the source or sets up a long-running connection to process events as soon as they arrive. Triggered Execution: The system starts, processes "what is currently available" (the delta between the last checkpoint and now), and then shuts down. While Serverless compute eliminates the need to manage infrastructure and can start up faster, selecting Serverless alone does not determine when the pipeline stops. If a Serverless pipeline is set to "Continuous" mode, it will still incur charges for maintaining the active state to listen for incoming data.

Question: A Lakeflow Spark Declarative Pipelines pipeline cleans web logs in the Silver layer. The requirement is:

Options:
  A: records with user_id IS NULL must be excluded from the target dataset
  B: the pipeline must continue running
  C: the number of excluded records must be tracked in pipeline metrics

A core component of this framework is Data Quality Expectations, which allow engineers to define constraints directly on datasets.
